### 环境及配置

在企业开发环境中，项目中用到的一些服务端软件如：MySQL、Nacos配置中心、RabbitMQ消息队列等通常会安装在企业局域网的服务器中，开发人员去远程连接它们。在教学中我们在自己的电脑上安装虚拟机，虚拟机代表了企业局域网中的服务器。

黑马教程中给的是Centos7，这里使用Ubuntu。

### 基础工程搭建

学成在线使用 Maven 来进行项目的管理和构建。整个项目分为三大类工程：父工程、基础工程 和微服务工程。

![img](./pic/clip_image002.gif)

每一种类的工程都有不同的作用，下面是对其功能进行说明：

- 父工程

  - 对依赖包的版本进行管理 

  - 本身为Pom工程，对子工程进行聚合管理 

- 基础工程

  - 继承父类工程

  - 提供基础类库

  - 提供工具类库   

- 微服务工程

  - 分别从业务、技术方面划分模块，每个模块构建为一个微服务。

  - 每个微服务工程依赖基础工程，间接继承父工程。

  - 包括：内容管理服务、媒资管理服务、搜索服务、订单支付服务等。

### 在maven的pom.xml文件中，\<dependencies>和\<dependencyManagement>下的\<dependencies>有什么区别

在Maven的`pom.xml`文件中，`<dependencies>` 和 `<dependencyManagement>` 下的 `<dependencies>` 元素有不同的作用和影响范围：

1. `<dependencies>` 元素：
   - `<dependencies>` 元素用于指定项目的依赖项（即所需的外部库或模块）。在这个部分声明的依赖项将被项目直接使用，并将被传递性地包含在项目的构建路径中。
   - 如果没有在 `<dependencyManagement>` 中定义相同的依赖项，那么 `<dependencies>` 中的依赖项将从中央仓库或其他仓库中下载并添加到项目中。

示例：
```xml
<dependencies>
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>example-library</artifactId>
        <version>1.0.0</version>
    </dependency>
</dependencies>
```

2. `<dependencyManagement>` 元素：
   - `<dependencyManagement>` 元素用于集中管理项目中可能用到的依赖项的版本号。在这个部分声明的依赖项版本号将被项目中的其他模块继承并使用，但不会自动添加到项目的构建路径中。
   - 当多个子模块或项目需要使用相同的依赖项时，可以在 `<dependencyManagement>` 中定义一次，以确保它们使用相同的版本号。这有助于维护和协调依赖项的版本。

示例：
```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>example-library</artifactId>
            <version>1.0.0</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

总结：
- `<dependencies>` 用于声明项目直接使用的依赖项，会自动包含在项目的构建路径中。
- `<dependencyManagement>` 用于集中管理依赖项的版本号，不会自动包含在项目的构建路径中，但可供子模块或项目继承使用。

### pom中的打包类型

在Maven项目的pom.xml文件中，`<packaging>` 元素用于指定项目的打包类型，即最终构建输出的文件类型。`<packaging>` 的值可以是不同的，用于表示不同类型的Maven项目，常见的包括：

1. `jar`：表示构建一个可执行的JAR文件。这通常用于Java应用程序，包括Java类文件和依赖库，以便将应用程序打包成一个可执行的JAR文件。

2. `war`：表示构建一个Web应用程序归档文件（WAR文件）。这通常用于包含Web应用程序的所有内容，包括Servlet、JSP、HTML、CSS、JavaScript等。

3. `pom`：表示构建一个用于组织和管理多个子项目的POM（Project Object Model）。这种类型的项目通常不会生成可执行的输出文件，而是用于组织和管理其他子模块。

4. `ear`：表示构建一个Enterprise Archive（EAR）文件，用于打包Java EE应用程序，包括EJB、WAR、JAR等文件。

5. `bundle`：表示构建一个OSGi bundle，用于模块化和部署Java应用程序的一种方式。

`<packaging>pom</packaging>` 表示项目的打包类型为POM，这意味着这个项目本身不会生成可执行的输出文件，而是用于管理其他子模块或项目。这通常在多模块项目中使用，其中顶级POM用于管理子模块的依赖关系、构建配置等。
